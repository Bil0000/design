# 14 — Figma Push

Pushes the design system into Figma. Creates variables, styles,
component frames, and a style-guide overview page. Source of truth is
`tokens.json` + the component library generated by modules 07 / 08.

Requires the Figma MCP. The MCP gate runs first.

Inputs: `.design-engine/system/tokens.json` (and the matching
`tokens.figma.json`, `tokens.css`, components dir). Output: writes
to a Figma file (new or existing), updates `.design-engine.json`
with `figmaFileKey` + `lastSync`, writes a local push log to
`.design-engine/system/figma-push.<timestamp>.log`.

Two binding rules govern this module:

> **MCP gate first.** If the Figma MCP isn't connected, this module
> halts with explicit reconnect instructions. It never falls back.
>
> **Idempotent updates.** Re-pushing the same system to the same
> file updates existing variables/styles/components in place — it
> does NOT duplicate them. Names are the identity key.

---

## 1. MCP Availability Check

Same gate as module 13 §1, with one addition: this module also
needs a write-capable tool surface (one of: `use_figma`,
`run_plugin`, `create_new_file`, `apply_change`, or any tool that
executes Plugin API JS against the file).

### 1.1 Probe

```
1. List registered MCP servers.
2. Match tools that look like:
     use_figma · run_plugin · create_new_file · apply_change ·
     create_local_variable · figma.* · figma_*
3. Required: at least ONE write tool (use_figma or equivalent), AND
   either `create_new_file` (for Mode A) or a way to target an
   existing fileKey (Mode B).
4. If neither, halt with §1.2.
```

### 1.2 Halt message

```
Figma MCP write capability is not available. design-engine needs a
server that can run Plugin API JS (use_figma / run_plugin) against
your file.

To connect:
  1. Open Claude Code settings → MCP servers.
  2. Add a Figma MCP that supports plugin execution. (Common: the
     official Figma MCP, figma-developer-mcp, framelink-mcp.)
  3. Authorize with a Figma personal access token that has WRITE
     access to the file (not just read).
  4. Restart this session and run /design system push again.

Or skip Figma push:
  Push later by running /design system push when MCP is connected.
  The system files in .design-engine/system/ are unchanged.
```

Wait for reconnect or alternative direction. Don't proceed.

## 2. Activation

Run when:

- The user message contains "push to figma", "sync to figma",
  "publish to figma", "send to figma", "export to figma", or the
  router selected this module via the `push to figma` pattern.
- The system-preview's "Push to Figma" button copied the trigger
  command to clipboard and the user pasted it.
- A separate flag `--push-figma` was set on a `/design system`
  invocation.

Skip when none of the above apply.

## 3. Two Push Modes

### 3.1 Mode A — New file

When:
- `.design-engine.json` has no `figmaFileKey`.
- The user said "push to a new file" / "create a Figma file".

Procedure:

1. Confirm before creating:
   ```
   Create a new Figma file for {system.name}? [Y]es / [N]o / [E]xisting (paste URL).
   ```
2. On Y, call:
   ```
   newFile = mcp.create_new_file({
     name: "{brief.product || system.name} — Design System",
     team_id: <if discoverable>
   })
   fileKey = newFile.fileKey
   ```
3. Open the file (write-only is fine; the user opens visually after).
4. Continue with the §4 workflow against `fileKey`.
5. On completion, save `figmaFileKey = fileKey` to
   `.design-engine.json`.

### 3.2 Mode B — Existing file

When:
- `.design-engine.json` has `figmaFileKey` set.
- The user provided a Figma URL or raw file key.

Procedure:

1. Resolve `fileKey`:
   ```
   1. From the user's pasted URL (parser from module 13 §3.1).
   2. From .design-engine.json.figmaFileKey.
   ```
2. Confirm target if both differ:
   ```
   Push to which file?
     [1] Saved file: {savedKey}  ({savedFileName})
     [2] Pasted URL: {pastedKey} ({pastedFileName if discoverable})
     [3] Cancel
   ```
3. On selection, run a `get_metadata(fileKey)` probe to confirm the
   file exists and the token has write access. If not, halt with a
   one-line message + the §1.2 instructions.
4. Continue with the §4 workflow.
5. On completion, update `lastSync` in `.design-engine.json`. Don't
   change `figmaFileKey` (it was already correct).

### 3.3 Idempotency

In Mode B, the push uses the variable / style / component name as
the identity key. The Plugin API patterns in §5 always look up by
name first; if found, update the existing value. If not, create.

## 4. The Push Workflow — 6 Steps

Every step runs inside one or more `use_figma(plugin_js)` calls.
Long pushes are batched: at most ~50 mutations per batch to keep
each plugin call under the MCP timeout.

### 4.1 Step 1 — Color variables

Create the primitive ramps + semantic layer + status palette + chart
palette.

Variables collection layout:

```
Collection "design-engine"
  Mode "Light"
  Mode "Dark"

Group  primitive
  color/grey-0 .. grey-950          COLOR
  color/accent-50 .. accent-950     COLOR
  color/success-* / warning-* / error-* / info-*
  color/chart-1 .. chart-8

Group  semantic
  bg-base / bg-surface / bg-elevated / bg-overlay / bg-inset
  text-primary / text-secondary / text-tertiary / text-disabled / text-inverse / text-link
  border-subtle / border-default / border-strong / border-focus
  accent / accent-hover / accent-press / accent-subtle / accent-text
  success / success-subtle / success-text  ... (warning/error/info same)

Group  component
  button-bg-primary / button-text-primary / button-bg-primary-hover ... (full module 08 §3.1 set)
  input-bg / input-border / input-border-focus / input-border-error
  card-bg / card-border / card-bg-hover
  ... (every component-layer token from module 07 §1.4)
```

Per variable:
- Set `valuesByMode[lightMode] = oklch_to_rgb(<lightValue>)`
- Set `valuesByMode[darkMode]  = oklch_to_rgb(<darkValue>)`

Figma stores colors in 0–1 RGB. Convert oklch → sRGB → 0–1 floats at
push time. Round to 4 decimals to avoid floating-point churn.

Semantic + component variables are stored as ALIAS variables that
reference the matching primitive (or another semantic):

```
semantic.bg-base       → alias to primitive.color.grey-0  (light)
                       → alias to primitive.color.grey-950 (dark)
component.button-bg-primary → alias to semantic.accent
```

This preserves the 3-layer architecture inside Figma so designers
can change a primitive and watch the cascade.

### 4.2 Step 2 — Number variables

Spacing + radius + duration scales.

```
Group  spacing
  space-1 .. space-32    FLOAT (px)

Group  radius
  radius-none / sm / md / lg / xl / 2xl / 3xl / full

Group  motion
  duration-fast / normal / slow / deliberate    FLOAT (ms)
```

Single-mode (numbers don't usually need light/dark variants).

### 4.3 Step 3 — Text styles

One TextStyle per scale stop, per family slot.

```
display/2xl    Cal Sans 32 / 1.2 / -0.03em / 600
display/xl     Cal Sans 24 / 1.3 / -0.02em / 600
body/base      Inter 16 / 1.6 / -0.01em / 400
body/sm        Inter 14 / 1.5 / 0 / 400
body/xs        Inter 12 / 1.6 / 0.01em / 400
mono/sm        JetBrains Mono 13 / 1.5 / 0 / 400
mono/base      JetBrains Mono 14 / 1.5 / 0 / 400
... (every cell from the type scale × {display, body, mono})
```

Set `fontName.family` from the system's font slot, `fontSize` from
the scale stop, `lineHeight` and `letterSpacing` from the same record,
`textCase` per the role (e.g. label = uppercase). Skip combinations
the system doesn't use.

### 4.4 Step 4 — Effect styles

Shadow scale (light mode). Dark mode systems still create the styles
for cross-mode component frames, but mark them as `light only` in
their description so designers know not to use them on dark surfaces.

```
shadow-xs / shadow-sm / shadow-md / shadow-lg / shadow-xl
focus-ring        2px outline at accent (35% alpha), 2px offset
```

Each EffectStyle's `effects[]` mirrors the system's shadow tokens
(module 07 §1.9). Convert oklch black with alpha to RGBA.

### 4.5 Step 5 — Component frames

For each component in `.design-engine/system/components/` (from
module 08), build a Figma component with its variants.

```
For component "button":
  - Create a COMPONENT_SET frame named "design-engine/Button".
  - Add property: variant (primary | secondary | ghost | destructive).
  - Add property: size (sm | md | lg | icon).
  - Add property: state (default | hover | active | focus | disabled | loading).
  - For each cross-product (variant × size × state):
    - Create a COMPONENT inside the set.
    - Apply auto-layout matching the JSX structure.
    - Apply fills using the matching component-layer color variable.
    - Apply text styles per role (label uses body/sm).
    - Apply corner radius from the radius variable.
    - Apply effect style for focus state (focus-ring).
```

Layout fidelity rules:
- Padding values come from spacing variables (no raw numbers in
  auto-layout).
- Gap comes from spacing variables.
- Width / height: hug content for buttons; fixed at touch-target
  minimum for icon buttons (40 px default).
- Corner radius bound to a radius variable.

Don't render every variant by hand. The Plugin API loops over the
combination space.

### 4.6 Step 6 — Style-guide overview page

A new page named `design-engine — Overview` containing:

```
Section 1  System metadata
  - System name, version, ambition, mode, last synced
  - Origin (pre-built / from-codebase / from-clone / from-figma / custom)

Section 2  Color
  - Each ramp as a row of swatches (12 stops)
  - Semantic color grid with names + values
  - Contrast matrix (WCAG ratios)

Section 3  Typography
  - Specimen blocks (display, body, mono)
  - Full type scale rendered with sample copy

Section 4  Spacing
  - Visual bars per stop with values

Section 5  Radius
  - Sample squares per stop

Section 6  Shadows / elevation
  - Sample cards per shadow stop (light)
  - Border-elevation lifts (dark)

Section 7  Components
  - Inserted instances of each COMPONENT_SET (default state) so
    designers can preview the library at a glance

Section 8  Anti-patterns
  - One frame per "Never" rule from SKILL.md §6, with a red ✗ badge
    and a one-line caption
```

The overview page is regenerated on every push (it's a derived
artifact). Components and variables persist across pushes; only the
overview is wholesale replaced.

## 5. Plugin API JS Patterns

Inside `use_figma(plugin_js)` calls, follow these patterns. They run
in Figma's sandboxed plugin environment, so document.* DOM is not
available; only the Figma plugin globals.

### 5.1 Create / find a variable collection

```js
let collection = figma.variables.getLocalVariableCollections()
  .find(c => c.name === "design-engine");

if (!collection) {
  collection = figma.variables.createVariableCollection("design-engine");
  // Default mode is renamed; second mode added explicitly.
  collection.renameMode(collection.modes[0].modeId, "Light");
  collection.addMode("Dark");
}

const lightMode = collection.modes.find(m => m.name === "Light").modeId;
const darkMode  = collection.modes.find(m => m.name === "Dark").modeId;
```

### 5.2 Create / update a color variable

```js
function upsertColorVariable(name, lightRGB, darkRGB) {
  let v = figma.variables.getLocalVariables("COLOR")
    .find(x => x.name === name && x.variableCollectionId === collection.id);

  if (!v) {
    v = figma.variables.createVariable(name, collection, "COLOR");
  }
  v.setValueForMode(lightMode, { r: lightRGB[0], g: lightRGB[1], b: lightRGB[2], a: lightRGB[3] ?? 1 });
  v.setValueForMode(darkMode,  { r: darkRGB[0],  g: darkRGB[1],  b: darkRGB[2],  a: darkRGB[3]  ?? 1 });
  return v;
}
```

For an alias (semantic → primitive):

```js
function setAlias(variable, mode, targetVar) {
  variable.setValueForMode(mode, figma.variables.createVariableAlias(targetVar));
}
```

### 5.3 Create a number variable

```js
function upsertNumberVariable(name, value) {
  let v = figma.variables.getLocalVariables("FLOAT")
    .find(x => x.name === name);
  if (!v) v = figma.variables.createVariable(name, collection, "FLOAT");
  v.setValueForMode(lightMode, value);
  v.setValueForMode(darkMode,  value);
  return v;
}
```

### 5.4 Create / update a text style

```js
function upsertTextStyle(name, spec) {
  let s = figma.getLocalTextStyles().find(x => x.name === name);
  if (!s) s = figma.createTextStyle();
  s.name = name;
  s.fontName = { family: spec.family, style: spec.weight };
  s.fontSize = spec.size;
  s.lineHeight = { value: spec.lineHeight, unit: spec.lineHeightUnit ?? "PERCENT" };
  s.letterSpacing = { value: spec.tracking, unit: "PERCENT" };
  if (spec.textCase) s.textCase = spec.textCase;
  return s;
}
```

### 5.5 Create / update an effect style

```js
function upsertEffectStyle(name, effects) {
  let e = figma.getLocalEffectStyles().find(x => x.name === name);
  if (!e) e = figma.createEffectStyle();
  e.name = name;
  e.effects = effects;     // array of DropShadow / InnerShadow / Blur
  return e;
}
```

### 5.6 Frame with auto-layout + variable bindings

```js
function buildButtonFrame({ variant, size, state }) {
  const f = figma.createFrame();
  f.name = `Button / ${variant} / ${size} / ${state}`;
  f.layoutMode = "HORIZONTAL";
  f.primaryAxisSizingMode = "AUTO";
  f.counterAxisSizingMode = "AUTO";
  f.itemSpacing = SPACING.gap;
  f.paddingLeft = SPACING.px;
  f.paddingRight = SPACING.px;
  f.paddingTop = SPACING.py;
  f.paddingBottom = SPACING.py;
  f.cornerRadius = 0;     // bound below
  f.setBoundVariable("topLeftRadius", radiusVar.id);
  f.setBoundVariable("topRightRadius", radiusVar.id);
  f.setBoundVariable("bottomLeftRadius", radiusVar.id);
  f.setBoundVariable("bottomRightRadius", radiusVar.id);

  // Fill bound to a color variable
  f.fills = [{ type: "SOLID", color: { r: 0, g: 0, b: 0 }, boundVariables: { color: { id: bgVar.id, type: "VARIABLE_ALIAS" } } }];

  return f;
}
```

`setBoundVariable` is the binding mechanism for radius, padding, gap,
sizing. For fills / strokes, use `boundVariables` inside the fill
record.

### 5.7 Component sets (variants)

```js
function buildButtonComponentSet() {
  const variants = ["primary", "secondary", "ghost", "destructive"];
  const sizes    = ["sm", "md", "lg", "icon"];
  const states   = ["default", "hover", "active", "focus", "disabled", "loading"];

  const components = [];
  for (const variant of variants) {
    for (const size of sizes) {
      for (const state of states) {
        const c = figma.createComponent();
        c.name = `variant=${variant}, size=${size}, state=${state}`;
        c.appendChild(buildButtonFrame({ variant, size, state }));
        components.push(c);
      }
    }
  }
  const set = figma.combineAsVariants(components, figma.currentPage);
  set.name = "design-engine/Button";
  return set;
}
```

`figma.combineAsVariants(components, parent)` produces the
COMPONENT_SET that designers consume.

## 7. `.design-engine.json` Update

After a successful push:

```json
{
  "system": "<unchanged>",
  "version": "<bumped patch only if tokens changed before this push>",
  "stack": "<unchanged>",
  "mode": "<unchanged>",
  "figmaFileKey": "<set>",
  "lastSync": "<ISO of this push>",
  "...": "..."
}
```

`figmaFileKey` is set on first push (Mode A). `lastSync` updates on
every push. `version` is NOT bumped by push alone — push reflects the
current local version, doesn't author new changes.

## 8. Failure Handling

If the push fails midway:

1. The plugin batches that succeeded are not rolled back (Figma
   doesn't support transactions across batches).
2. The local push log records exactly which step + which item failed.
3. The user is told:

   ```
   Push paused at Step {N}: {what failed}.
   Successfully pushed: {summary up to failure}.
   Resume with: claude /design system push --resume
   ```

4. On `--resume`, the module reads the last log, reads the file's
   current state via `get_variable_defs` + `get_styles`, and only
   re-pushes items missing or out of date. (Idempotency from §3.3
   makes resume safe.)

`.design-engine.json` update + a local log. Updates the

```yaml
  push_at:           <ISO>
  file_key:          {KEY}
  variables_pushed:  <int>
  text_styles_pushed: <int>
  effect_styles_pushed: <int>
  components_pushed: <int>

  log_path:          .design-engine/system/figma-push.<ts>.log
```

Hand off to:
- module 16 (handoff) when `/design system push` was the active

- The orchestrator returns to the user on a standalone push with
  the §6 confirmation as the final message.
