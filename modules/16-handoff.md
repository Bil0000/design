# 16 — Handoff

The final and most important output. Bundles every artifact a coding
AI (Claude Code, Cursor, Codex, Aider, Continue) needs to implement
the design pixel-perfect — with zero clarifying questions back to the
designer.

Inputs: everything in `.design-engine/system/`,
`.design-engine/canvas/<latest>/`, `.design-engine/prototype/<latest>/`,
`.design-engine/critique.json`. Output dir:
`.design-engine/handoff/`.

The binding rule for this module:

> **The handoff is self-sufficient.** Any AI agent given only
> `HANDOFF.md` + the bundled screens + tokens must be able to ship
> the design without further input from the designer. If the agent
> would have to ask a question, the answer belongs in HANDOFF.md.

---

## 1. Activation

Run when:
- Critique gate is `approved` or `warnings`.
- Critique gate is `blocked` AND the user passed `--acknowledge`.
- The active command is anything except `/design review` (review
  produces no handoff).

Skip otherwise.

## FOR THE AI READING THIS

You are implementing this design. Read this file end-to-end before
writing code. Then implement screens in the order specified. Reference
the screenshot in `screens/<NN>-<id>.png` for pixel accuracy.

**Hard contract:**

1. Use the design tokens. Never hardcode color, spacing, radius,
   shadow, font-size, line-height, tracking, duration, or easing.
   Tokens are at `design-tokens.css` / `design-tokens.tailwind.js`
   in this directory.

2. Reuse the bundled components in `components/`. Do not rebuild
   Button, Input, Card, Badge, Nav, Table, Modal, Toast, Avatar,
   Select, Switch, Checkbox, Radio, Skeleton. Compose, don't copy.

3. Match the screenshots. The screen HTML in `screens/<NN>-<id>.html`
   is your structural reference; the PNG is the visual reference.
   When in doubt, the PNG wins.

4. Ship every state. Hover, focus-visible, active, disabled. Empty,
   loading, error. The screen specs (§ Per-Screen Specs) list which
   states each screen must support.

5. Accessibility is required, not optional. Every interactive
   element has a real semantic element (button / a / input), an ARIA
   label where ambiguous, full keyboard navigation, and a visible
   focus ring at `var(--border-focus)`.

6. Dark mode is built in. Toggle via the `.dark` class (or
   `[data-theme="dark"]`) on `<html>`. Don't fork components.

7. Animation: `transition: transform var(--duration-normal) var(--easing-out),
   opacity var(--duration-fast) var(--easing-out)`. Never animate
   `box-shadow`, `width`, `height`, `top`, `left`. Honor
   `prefers-reduced-motion: reduce`.

8. Copy in the final UI matches the copy in the screenshots. Don't
   substitute Lorem ipsum, "Click me", or boilerplate.

If a question would normally need to be asked, the answer is
somewhere below. Re-read before asking.

## DESIGN TOKENS

Tokens are the source of truth. Every color, spacing, radius, font,
duration, and easing in this design IS a token. Do not bypass.

**Import paths (after promotion):**

```css
/* in your global stylesheet */
@import "./styles/tokens.css";
```

```ts
// in TypeScript
import { tokens } from "@/styles/tokens";
```

```js
// tailwind.config.js
const tokens = require("./styles/tokens.tailwind");
module.exports = { theme: { extend: tokens.theme.extend } };
```

**Never-hardcode rule (binding):**

| Property        | Use                                     |
|-----------------|------------------------------------------|
| Color           | `var(--bg-base)` / `bg-base` class       |
| Spacing         | `var(--space-4)` / `p-4`                 |
| Radius          | `var(--radius-md)` / `rounded-md`        |
| Font size       | `var(--text-base)` / `text-base`         |
| Line height     | embedded in the type token via `text-*`  |
| Tracking        | embedded in the type token               |
| Shadow          | `shadow-md` (light only)                 |
| Duration        | `duration-fast` / `duration-normal`      |
| Easing          | `ease-out` / `ease-in`                   |

If you find yourself reaching for a literal `#hex`, `Npx`, or
`Nms`, stop — there is a token. Open `design-tokens.css` and find it.

## TYPOGRAPHY RULES

Fonts:
```
display:  {family} — for headings ≥ 24px only
body:     {family} — for everything else, including chrome text
mono:     {family} — for code, IDs, tabular numbers, version strings
```

Weights loaded: `{weights}` — do not introduce new weights.

Scale (every stop, with the line-height + tracking + weight that go
with each):

```
xs       12px / 1.6  / +0.01em / 400
sm       14px / 1.5  / 0       / 400
base     16px / 1.6  / -0.01em / 400
lg       20px / 1.4  / -0.02em / 500
xl       24px / 1.3  / -0.02em / 600
2xl      32px / 1.2  / -0.03em / 600
3xl      40px / 1.1  / -0.04em / 700
4xl      56px / 1.0  / -0.05em / 700
```

Hard rules:
- Maximum 3 different sizes per screen.
- Maximum 2 weights in any single visual group.
- Negative tracking on every size ≥ 20px.
- Display font only on sizes ≥ 24px.
- Mono only for code / IDs / tabular numbers / version strings.
- Body font for chrome text, including small labels.

---

## SPACING SYSTEM

Base unit: {N}px. Scale: {1, 2, 3, 4, 6, 8, 12, 16, 24, 32}.

Allowed values are the scale. Never use arbitrary spacing
(`p-[17px]`, `gap-[14px]`). When the design appears to require an
off-stop value, snap to the nearest stop. When that's wrong, file
back as a system change — not a one-off.

Named aliases:
```
gutter-sm    space-3     12px
gutter-md    space-4     16px
gutter-lg    space-6     24px
gutter-xl    space-8     32px
section-y    space-16    64px
```

## PER-SCREEN SPECS

For every screen there is a spec block here, plus a sibling files in
`screens/`:

```
screens/<NN>-<id>.png         visual reference
screens/<NN>-<id>.html        structural reference
screens/<NN>-<id>.notes.md    measurements + per-element notes
screens/<NN>-<id>.spec.json   machine-readable bbox / token map
```

Spec block format (one per screen):

### Screen 1 — Dashboard

**Visual reference:** `screens/01-dashboard.png`
**HTML reference:**   `screens/01-dashboard.html`
**Frame:**            web · 1440 × 900 (light + dark)

**Layout**

- 240px sidebar (left), main content fills remaining width.
- Sidebar uses `Nav.Sidebar`. Active item indicated by a 2px-wide
  accent bar absolutely positioned at the left edge.
- Main content: `gap-6` between primary regions, `space-8` outer
  padding.
- Header strip in main: KPI tiles row (`grid-cols-4`, `gap-4`) above
  the activity table.

**Measurements (key)**

- Sidebar items: `h-9 px-3` (36 × full width)
- KPI tile: `p-5 rounded-lg border border-subtle`
- Activity table: full width, `rounded-lg overflow-hidden`,
  sticky header, virtualization required at >200 rows.
- CTA "Create new": Button (variant=primary, size=md), top-right of
  header strip.

**Components used**

- `Nav.Sidebar` — items: Dashboard, Projects, Reports, Settings, Billing.
- `Card` (×4) — KPI tiles. Variant: `default`. Each renders a label,
  a numeric value, and a delta indicator.
- `Table` — columns: name, status, owner, updated_at, actions.
  Sortable: name, updated_at. Selectable: yes.
- `Badge` — status column (success / warning / error / default).
- `Avatar` — owner column, size sm.
- `Button` — header CTA + bulk-action toolbar.

**Interactions**

- Sidebar item hover: bg-surface, 100ms ease-out.
- Sidebar item active: text-primary + bg-elevated + accent indicator
  bar (animated translateX between items, never re-rendering).
- Table row hover: bg-surface.
- Table row click: navigates to the project detail (route:
  `projects/<id>` — outside this delivery; treat as a no-op for now).
- Sort header click: re-sort + animate the sort caret.
- "Create new" click: opens the new-project modal (see screen 7
  modal — same pattern).

**States**

- default · empty (see screen 2) · loading (skeleton tiles + skeleton
  rows for table) · error (top-of-page banner using `Toast`).

**A11y**

- Sidebar uses `<nav aria-label="Primary">` containing `<ul role="list">`.
- Active item gets `aria-current="page"`.
- Sort buttons inside `<th>` set `aria-sort="ascending|descending|none"`.
- Bulk-select checkbox sets `aria-checked="mixed"` when partially
  selected.
- Focus order: sidebar → CTA → KPI tiles → table header → first row.

**DO NOTs**

- Do not animate the active indicator with opacity / multiple
  re-renders. One absolutely-positioned bar, translateX only.
- Do not add a search input to the header. Not in this screen's
  scope.
- Do not change the KPI tile order (the order encodes the dashboard's
  read priority — see the visual reference).
- Do not introduce a different shadow strategy in dark mode. Dark
  mode uses border-elevation; the system already encodes this.

## VERIFICATION CHECKLIST

Use this checklist as you finish each screen. Don't mark a screen
done until every box is checked.

```
□ Matches the screenshot at 1× and 2× DPI
□ Light + dark modes both render correctly
□ All states implemented: default, hover, focus-visible, active,
  disabled, loading, empty, error (where applicable)
□ Keyboard navigation reaches every interactive element
□ Focus ring visible on Tab, hidden on click (focus-visible)
□ ARIA roles + labels present for non-semantic elements
□ Touch targets ≥ 44 × 44 (mobile flows only)
□ No hardcoded color / spacing / radius / shadow / duration values
  (audit: grep '#[0-9a-fA-F]\{3,8\}' should return zero hits)
□ No animation on box-shadow / width / height / top / left
□ prefers-reduced-motion respected (animations cut to 0 except
  entrance fades)
□ WCAG AA contrast on all text/bg pairings
□ Real content used (no Lorem ipsum, no "Click me", no placeholder)
□ Empty state designed (not just a "No data" string)
□ Error state designed (with a clear next action)
□ Loading state uses Skeleton, not blank
□ Distinctive detail from the spec is implemented (the screenshottable
  moment)
□ Component imports point at the bundled `components/` (or your
  project's promoted path)
□ TypeScript types are explicit (no `any` in props)
□ Lighthouse a11y score ≥ 95 on the screen
```
