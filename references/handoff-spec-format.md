# Handoff Spec Format

Reference for module 16 (handoff). Defines exactly how to write a
HANDOFF.md the implementing AI can execute from without asking
clarifying questions back.

The audience is AI coding agents — Claude Code, Cursor, Codex, Aider,
Continue. They follow text precisely. They cannot see your design
in your head. They CAN see the bundled `screens/*.png`, and they read
your prose literally.

---

## 1. Philosophy — Precision Over Brevity

A handoff is not a designer's note. It's a contract.

Every word the AI reads becomes a constraint or a license. Vague
prose becomes a license to drift. Specific prose becomes a constraint
that lands the design.

### The four contracts

```
PRECISION         Every value is exact. No "roughly", "about",
                  "around", "small", "medium", "large".

UNAMBIGUITY       Every instruction has one possible interpretation.
                  No "use your judgment", "as appropriate", "where
                  it makes sense".

EXHAUSTIVENESS    Every state of every interactive element is
                  documented. Default + hover + focus + active +
                  disabled + loading + empty + error.

VERIFIABILITY     Every claim is checkable. "≥ 4.5 contrast",
                  "44 × 44 minimum", "100ms transition", "max-width
                  1280px". Numbers, not adjectives.
```

If you find yourself writing something an implementing AI could
"interpret", rewrite it as something they have to obey.

### Length follows fidelity

A handoff for one screen at ambition A might be 200 lines. A handoff
for an 8-screen system at ambition C is 2000+ lines. Don't try to
shorten the contract — the cost of a 30-line section is one prompt.
The cost of a wrong implementation is a re-do.

---

## 2. The Complete HANDOFF.md Template

All sections. Every one is mandatory unless explicitly marked
optional. Module 16 §3 is the canonical version of this template;
this reference adds writing rules + examples.

```markdown
# HANDOFF — {brief.product || command}

*Generated by design-engine · {ISO timestamp}*
*System: {system.name} v{system.version} · ambition {A|B|C|D}*

## FOR THE AI READING THIS
[8-point hard contract — see module 16 §3]

## STACK
[exact versions of every dependency the design assumes]

## DESIGN TOKENS
[import paths + never-hardcode rule + token quick-reference table]

## TYPOGRAPHY RULES
[fonts + scale + hard rules]

## SPACING SYSTEM
[base unit + scale + named aliases + snap rule]

## SCREEN IMPLEMENTATION ORDER
[numbered table with dependencies + suggested output paths]

## PER-SCREEN SPECS
[one canonical block per screen — §3.4 below]

## COMPONENTS TO BUILD
[new components only, or "None — bundled cover this"]

## WHAT'S ALREADY BUILT — DON'T REBUILD
[bundled component list with paths]

## VERIFICATION CHECKLIST
[checkable boxes per screen]

## RECORDED WARNINGS / DEFECTS
[only if critique gate was warnings or --acknowledge]

## ANIMATIONS
[only if module 11 ran]

## CHANGELOG
[append-only, every regeneration logs here]

## CONTACT POINTS
[pointers to SYSTEM.md / tokens / components / critique / verification]
```

## 3. Per-Section Writing Rules

### 3.1 FOR THE AI READING THIS — the contract

Eight numbered rules. Each is one paragraph. Imperative voice. State
the WHAT and the consequence.

Bad:
```
Try to use the tokens when you can.
```

Good:
```
Use the design tokens. Never hardcode color, spacing, radius, shadow,
font-size, line-height, tracking, duration, or easing. Tokens are at
`design-tokens.css` / `design-tokens.tailwind.js` in this directory.
```

Rules:
- No "should" / "try to" / "where possible" — use "must" / "do" /
  imperative.
- Each rule names the artifact the AI consults.
- Each rule states the consequence implicitly (the contract is the
  consequence).

### 3.2 STACK — exact versions

```
✗ "Use React"
✓ "Framework: React 18.2.0 (via Next.js 14.2.0 App Router)"

✗ "Tailwind"
✓ "Styling: Tailwind v3.4.0 with v4 @theme syntax disabled"

✗ "Some component library"
✓ "Component lib: shadcn/ui (style: new-york), Radix primitives 1.1.x"
```

The implementing AI must know exactly what's installed. Versions
matter — Tailwind v3 vs v4 is a different config file. Next.js 13
vs 14 is a different routing model.

### 3.3 DESIGN TOKENS — import paths + never-hardcode

Show three things:

1. The exact import statements the AI should add to the project.
2. The token-property quick-reference table mapping CSS property →
   token usage form.
3. The hardcode prohibition with grep verification.

Example for the table:

```markdown
| Property        | Use                                     |
|-----------------|------------------------------------------|
| Color           | `var(--bg-base)` / `bg-base` class       |
| Spacing         | `var(--space-4)` / `p-4`                 |
| Radius          | `var(--radius-md)` / `rounded-md`        |
| Font size       | `var(--text-base)` / `text-base`         |
| Shadow          | `shadow-md` (light only)                 |
| Duration        | `duration-fast` / `duration-normal`      |
| Easing          | `ease-out` / `ease-in`                   |
```

Don't paste the entire token JSON here. The AI has the file. Tell it
the file exists; let it open it.

### 3.4 PER-SCREEN SPEC — the canonical block

This is the bulk of the document and the place handoffs go wrong.
The block has 7 sub-sections, in this exact order:

```
### Screen N — {label}

**Visual reference:** `screens/<NN>-<id>.png`
**HTML reference:**   `screens/<NN>-<id>.html`
**Frame:**            web · 1440 × 900 (light + dark)

**Layout**
[Layout description — see §3.4.1]

**Measurements (key)**
[Measurements — see §3.4.2]

**Components used**
[Component list with prop values — see §3.4.3]

**Interactions**
[Interactions with timing + easing — see §3.4.4]

**States**
[State enumeration — see §3.4.5]

**A11y**
[Accessibility requirements — see §3.4.6]

**DO NOTs**
[Specific anti-patterns for THIS screen — see §3.4.7]
```

#### 3.4.1 Layout — describe structure, not aesthetics

Bad:
```
A clean dashboard with a sidebar on the left and content on the right.
```

Good:
```
- 240px sidebar (left, sticky), main content fills remaining width.
- Main content padding: 32px outer.
- Header strip in main: 64px tall, gap-6 from KPI grid.
- KPI grid: 4 columns, gap-4 (16px).
- Below KPI: activity table at full width, sticky header.
```

Rules:
- Use exact px or token names. Never "large", "medium", "small".
- Describe layout as a tree of containers, not as visual composition.
- Always specify width + height + padding + gap on every container.

#### 3.4.2 Measurements — px or token references

Always one of:

```
240px           ← when no token fits
space-4 (16px)  ← when a token fits, name token + value
h-9 (36px)      ← when a Tailwind class fits, name class + value
```

Never:
```
roughly 16px
about 1rem
medium gap
```

Worked example:

```markdown
**Measurements (key)**

- Sidebar items: `h-9 px-3` (36 × full sidebar width)
- KPI tile: `p-5 rounded-lg border border-subtle`
- KPI tile label: `text-xs uppercase tracking-wider text-secondary`
- KPI tile value: `text-2xl font-semibold text-primary`
- KPI tile delta: `text-xs text-success` (positive) or `text-error` (negative)
- Activity table: full width, `rounded-lg overflow-hidden`, sticky header
- Activity row height: 52px (`h-13`)
- "Create new" button: Button (variant=primary, size=md), top-right of header strip
```

Every line is checkable.

#### 3.4.3 Components used — tree with prop values

Bad:
```
Uses our Card component and a Button somewhere.
```

Good:
```
**Components used**

- `Nav.Sidebar` — items: Dashboard, Projects, Reports, Settings, Billing.
  Active item: derived from current route. Collapsed prop: respects
  user preference (localStorage key `sidebar.collapsed`).

- `Card` (×4, in a 4-col grid) — KPI tiles. Variant: `default`. Each:
  - label  (text-xs uppercase tracking-wider text-secondary)
  - value  (text-2xl font-semibold text-primary, font-feature-settings: 'tnum')
  - delta  (text-xs, color from variant: success/warning/error/secondary)

- `Table` — columns: name, status, owner, updated_at, actions.
  Sortable: name, updated_at. Selectable: yes (multi-select).
  Page size: 25.

- `Badge` — used in the Status column. Variant per row: success | warning | error | default.

- `Avatar` — used in the Owner column, size sm.

- `Button` — header CTA + bulk-action toolbar (when rows selected).
  variant=primary size=md for CTA; variant=ghost size=sm for toolbar items.
```

Rules:
- Every component has a path or a name from the bundled set.
- Every prop the AI must pass is named.
- Per-instance overrides are listed explicitly.

#### 3.4.4 Interactions — always timing + easing

Bad:
```
Sidebar items have a hover state. Buttons have a press feedback.
Modal opens with animation.
```

Good:
```
**Interactions**

- Sidebar item hover: bg-surface, transition `background-color 100ms ease-out`.
- Sidebar item active: text-primary + bg-elevated + accent indicator
  bar. The bar is `2px wide`, `absolutely positioned at left: 0, top: 6px,
  bottom: 6px`. Animate the bar's `transform: translateY` between items
  (don't re-render). Duration: `150ms ease-out`.
- Table row hover: bg-surface, `100ms ease-out`.
- Table row click: navigate to `/projects/<id>`. Out of scope for this
  handoff; treat as `console.log` placeholder.
- Sort header click: re-sort + animate the sort caret rotation
  (180deg) using `transform 150ms ease-out`.
- "Create new" click: opens the new-project modal (Modal component,
  size=md). Trigger transition is the standard Modal pattern from
  components/modal.tsx.
- Bulk-action toolbar appears when ≥ 1 row is selected. Slide down
  from top of table, `transform: translateY 200ms ease-out` + fade-in
  `opacity 100ms ease-out`.
- prefers-reduced-motion: cut all `translateY` to 0; keep fades.
```

Every interaction is one of:
- A token-referenced transition (`100ms ease-out`).
- A keyboard shortcut.
- A specific user gesture with explicit consequence.

#### 3.4.5 States — enumerate all

```
**States**

- default · empty · loading · error
- (interactive) hover · focus-visible · active · disabled

For this screen:
- empty:   No projects → empty state from screens/02-dashboard-empty.png
- loading: skeleton tiles for KPI (4 × Skeleton variant=block) + 5
           skeleton rows for table (Skeleton variant=line)
- error:   inline Toast (variant=error) at top of main content,
           explaining the failure with a Retry action
```

Don't list states without saying what they look like + which artifact
shows them.

#### 3.4.6 A11y — role, ARIA, keyboard

Use the format:

```
**A11y**

- Element type: <element-name> uses <tag-or-role>.
  - aria-* attributes: <list>.
  - keyboard: <shortcut → effect>.
- Focus order: <ordered list of focusable elements>.
- Live regions: <if any, with aria-live + role>.
```

Worked example:

```markdown
**A11y**

- Sidebar uses `<nav aria-label="Primary">` containing `<ul role="list">`.
- Each NavItem is an `<a>` with `aria-current="page"` when active.
- Sort buttons inside `<th>` are `<button>` with `aria-sort="ascending|descending|none"`.
- Bulk-select header checkbox uses `aria-checked="mixed"` when partially selected.
- Row checkbox: `aria-label={"Select row " + index}`.
- Focus order: sidebar → "Create new" CTA → KPI tiles (4) → table header sort buttons → first row checkbox → first row content → pagination prev → pagination next.
- Live regions:
  - Toast container: `role="status" aria-live="polite"` for default/info/success.
  - Toast container (error): `role="alert" aria-live="assertive"`.
- Keyboard:
  - `j/k` — move row focus down/up (when table is focused).
  - `Enter` — open the focused row (same as click).
  - `x` — toggle row selection.
  - `?` — open keyboard shortcuts modal.
```

Every interactive element has its semantic element + ARIA + keyboard
contract listed. Don't say "make it accessible" — list the requirements.

#### 3.4.7 DO NOTs — specific to this screen

Generic "don't use shadows in dark mode" lives in the system anti-
patterns. The DO NOTs section names the traps THIS screen is at risk
of falling into.

Bad:
```
Don't make it look generic.
```

Good:
```
**DO NOTs**

- Do not animate the active sidebar indicator with opacity. One
  absolutely-positioned bar, translateY between items only.
- Do not add a search input to the header. Out of scope.
- Do not change the KPI tile order. The order encodes the dashboard's
  read priority: Active projects → Issues → Latency → Uptime.
- Do not introduce shadows in dark mode. The system uses border
  elevation (bg-base / bg-surface / bg-elevated ladder).
- Do not use shadcn's default badge styles. Use the bundled Badge
  component which respects status-subtle / status-text token pairs.
- Do not center-align the table cells. Default left, except numeric
  columns (right-align).
- Do not load > 25 rows at once. Pagination required.
```

Rules:
- Every DO NOT names a thing the AI might do otherwise.
- Specific enough that the AI can recognize the failure mode.
- Tied to the screen's particulars, not generic advice.

## 4. Screenshot Reference System

The bundled `screens/<NN>-<id>.png` is the visual oracle. When prose
and screenshot disagree, the screenshot wins (because the design was
approved at the canvas stage).

### 4.1 Path conventions

```
screens/01-dashboard.png            default mode (system-default)
screens/01-dashboard-light.png      light mode (when system has both)
screens/01-dashboard-dark.png       dark mode (when system has both)
screens/01-dashboard.html           the source HTML (structural reference)
screens/01-dashboard.notes.md       per-element measurements + tokens
screens/01-dashboard.spec.json      machine-readable bbox + token map
```

### 4.2 How to reference in prose

Always use the path relative to the handoff root. Always state which
mode if ambiguous:

```
✗ See the screenshot
✓ Visual reference: `screens/01-dashboard.png`
✓ Visual reference: `screens/01-dashboard-light.png` (light mode primary)
                    `screens/01-dashboard-dark.png` (dark mode equivalent)
```

### 4.3 The per-element notes file

`screens/<NN>-<id>.notes.md` is a sibling, not embedded in HANDOFF.md.
Its job is element-by-element measurements + token attributions —
the kind of detail that would bloat HANDOFF.md if inlined.

The implementing AI is expected to:
1. Read HANDOFF.md end-to-end.
2. For each screen it implements, open the matching `notes.md`.
3. Reference the `spec.json` when needing exact bbox / token data
   programmatically.

### 4.4 The spec.json contract

```json
{
  "screen_id": "dashboard",
  "viewport": { "w": 1440, "h": 900 },
  "elements": [
    {
      "selector": ".dashboard__cta",
      "component": "Button",
      "variant": "primary",
      "size": "md",
      "bbox": { "x": 1276, "y": 32, "w": 132, "h": 40 },
      "tokens": {
        "bg":       "bg-accent",
        "text":     "accent-text",
        "radius":   "radius-md",
        "duration": "duration-fast",
        "easing":   "easing-out"
      }
    }
  ]
}
```

The implementing AI uses this when checking pixel accuracy. Modules
that visually verify (e.g. an automated screenshot diff agent) read
this directly.

## 5. Common Handoff Mistakes

The patterns that cause AIs to deviate. Each is a mistake the
generating AI made — not the implementing AI's fault.

### 5.1 The "use your judgment" license

Bad:
```
Spacing should feel comfortable.
```

The implementing AI now invents spacing. Drift starts here.

Good:
```
Spacing: outer 32px, inner gap-4 (16px), tile gap-3 (12px).
```

### 5.2 The "where appropriate" license

Bad:
```
Add hover states where appropriate.
```

"Where appropriate" is "everywhere the AI guesses". Half come out
right, half wrong.

Good:
```
Hover states required on: every Button, every NavItem, every Table
row, every interactive Card, every link in body copy. Use `bg-surface`
shift, transition `100ms ease-out`.
```

### 5.3 Vague color references

Bad:
```
Use a slightly darker grey for the secondary text.
```

Multiple greys exist in the system. "Slightly darker" is unspecified.

Good:
```
Secondary text: text-secondary token (resolves to grey-400 in dark,
grey-600 in light).
```

### 5.4 Unnamed components

Bad:
```
Build a card with a title, image, and button.
```

Which card? Existing Card composed how?

Good:
```
Use bundled Card (default variant) with:
- CardHeader containing the title (text-lg, font-semibold)
- CardMedia containing the image (aspect-ratio 16/9)
- CardBody containing the description (text-sm, text-secondary)
- CardFooter containing one Button (variant=primary, size=md)
```

### 5.5 Missing states

Bad:
```
The form has fields and a submit button.
```

What about: empty state, partial-fill state, validation in progress,
post-submit loading, success, network error, validation errors per
field?

Good: list every state explicitly with what triggers it and what the
visual is. Reference screenshots for each.

### 5.6 Unverifiable a11y notes

Bad:
```
Make the page accessible.
```

"Accessible" is the name of the goal, not a contract.

Good: list every interactive's role, ARIA attributes, keyboard
shortcut, and the focus order of the screen.

### 5.7 Inconsistent terminology

Bad:
```
The header bar (also called nav-bar, top-bar, or app bar) ...
```

Three names for one thing. The AI now uses three different
identifiers in code.

Good: pick one name. "Topbar". Use it everywhere. Define it once in
the components-used list.

### 5.8 Over-loose copy slots

Bad:
```
Add some marketing copy.
```

The AI fills in Lorem ipsum or invents copy that doesn't fit.

Good: list every copy slot with the actual copy or a `{{ slot.name }}`
template variable + format constraints (max chars, casing).

### 5.9 Reversing the screenshot contract

Bad:
```
The hero section should be a bit different from the screenshot —
maybe try a slightly different layout.
```

If the screenshot is right, write the screenshot's layout. If it's
wrong, regenerate the screenshot. Don't ship a handoff where prose
fights image.

### 5.10 Mixing system rules with screen rules

Bad:
```
Don't use shadows in dark mode (system rule), don't change the KPI
order on this screen (screen rule), use real images (system rule).
```

Generic system rules belong in §FOR THE AI READING THIS or in the
linked SYSTEM.md. Screen-specific DO NOTs belong in the per-screen
DO NOTs. Mixing them dilutes both.

## 6. Complete Example — Dashboard Screen

A full, real-world per-screen block. Long, but every line earns its
place.

```markdown
### Screen 1 — Dashboard

**Visual reference:** `screens/01-dashboard.png`
**HTML reference:**   `screens/01-dashboard.html`
**Frame:**            web · 1440 × 900 (light + dark variants present)
**Source canvas:**    .design-engine/canvas/2026-04-29T18-22-00/

**Purpose**

The home screen for the analytics product. Default route after auth.
Shows project KPI rollups, the recent activity table, and primary
"Create new" action. Power users live here all day; density is
deliberate.

**Layout**

- Outer container: `min-h-screen` flex row.
- Left: `Nav.Sidebar` at 240px (collapsed: 64px). Sticky `top-0`,
  `h-screen overflow-y-auto`, `border-r border-subtle`.
- Right: main content area, `flex-1 p-8` (32px outer padding).
- Header strip (inside main): flex row, `h-16` (64px), `items-center
  justify-between`, `mb-6` (24px).
  - Left of strip: page title + breadcrumb.
  - Right of strip: "Create new" Button + secondary "Filter" Button.
- KPI grid: `grid grid-cols-4 gap-4 mb-8` (16px gap, 32px below).
- Activity table: full width, `mb-6`.
- Footer (inside main, below table): pagination row + bulk-action
  toolbar (slides down when rows are selected — see Interactions).

Mobile (≤ 768px): sidebar collapses to a Drawer; KPI grid becomes
2-col then 1-col.

**Measurements (key)**

- Sidebar: `w-60` (240px). Items: `h-9 px-3` (36px × full width).
  Section labels: `text-xs uppercase tracking-wider text-tertiary
  px-2 pt-3 pb-1`.
- Header strip:
  - Page title: `text-2xl font-semibold text-primary`.
  - Breadcrumb: `text-sm text-secondary`, `ml-3`.
  - "Create new" Button: variant=primary, size=md (40px tall).
  - "Filter" Button: variant=ghost, size=md, leftIcon=<FilterIcon />.
- KPI tile (4 of them, in `grid-cols-4`):
  - Container: `p-5 rounded-lg border border-subtle bg-surface`.
  - Label row: `text-xs uppercase tracking-wider text-secondary mb-2`.
  - Value: `text-2xl font-semibold text-primary` with
    `font-variant-numeric: tabular-nums`.
  - Delta: `text-xs mt-1`, color: success-text (up) | error-text
    (down) | text-secondary (unchanged).
  - KPI order (do not change): Active projects → Issues → Latency
    p95 → Uptime.
- Activity table:
  - Container: `rounded-lg overflow-hidden border border-subtle`.
  - Header row: `bg-surface`, `text-xs uppercase tracking-wider
    text-secondary`, `h-10`.
  - Body row: `bg-base`, hover `bg-surface`, `h-13` (52px).
  - Cell padding: `px-4 py-3`.
  - Cell border bottom: `border-b border-subtle`.
  - Columns + widths: name (32%) · status (12%) · owner (16%) ·
    updated_at (16%) · actions (24%).
- Pagination row: `text-sm text-secondary`, ghost buttons.

**Components used**

- `Nav.Sidebar` (from components/nav.tsx)
  - `items`: 5 sections.
    [
      { label: "WORKSPACE", items: [
        { label: "Dashboard",  href: "/",          icon: <DashboardIcon/>, active },
        { label: "Projects",   href: "/projects",  icon: <ProjectsIcon/> },
        { label: "Reports",    href: "/reports",   icon: <ReportsIcon/> }
      ]},
      { label: "TEAM", items: [
        { label: "Members",    href: "/team",      icon: <MembersIcon/> },
        { label: "Settings",   href: "/settings",  icon: <SettingsIcon/> },
        { label: "Billing",    href: "/billing",   icon: <BillingIcon/>, badge: 3 }
      ]}
    ]
  - `collapsed`: respects localStorage key `sidebar.collapsed`. Default: false.

- `Button` (header strip CTA)
  - variant=primary, size=md, leftIcon=<PlusIcon w=14 h=14 />
  - onClick: opens the "New project" Modal (size=md).

- `Button` (header strip secondary)
  - variant=ghost, size=md, leftIcon=<FilterIcon />
  - onClick: opens a Popover with filter options (out of scope for
    this screen — render the Button only).

- `Card` (×4, KPI tiles, default variant)
  - props: variant=default
  - children:
    - <div className="text-xs uppercase tracking-wider text-secondary mb-2">{label}</div>
    - <div className="text-2xl font-semibold text-primary tabular-nums">{value}</div>
    - <div className={`text-xs mt-1 ${deltaClass}`}>{delta}</div>

- `Table` (from components/table.tsx)
  - columns: see §Measurements
  - data: from props (server-fetched in real impl; static array
    in this implementation)
  - selectable: true
  - pageSize: 25
  - onRowClick: navigate to `/projects/<id>`. For this delivery,
    `console.log('row click', id)`.
  - onSelectionChange: shows the bulk-action toolbar (see Interactions).

- `Badge` (used in Status column)
  - per-row variant from `row.status`:
      "active" → variant=success
      "blocked" → variant=warning
      "failed" → variant=error
      otherwise → variant=default

- `Avatar` (used in Owner column)
  - size=sm, alt={owner.name}, src={owner.avatarUrl}

- `Modal` (new project) — triggered by header CTA
  - size=md
  - title: "Create new project"
  - body: a 3-field form (name, description, visibility). Submit
    button is variant=primary; cancel is variant=ghost.

**Interactions**

- Sidebar item hover: `transition: background-color var(--duration-fast) var(--easing-out)`.
- Sidebar item active: text-primary + bg-elevated + 2px accent
  indicator bar at `left: 0`, `top: 6px`, `bottom: 6px`. The indicator
  is one persistent element; on active-item change, animate
  `transform: translateY(<delta>) var(--duration-normal) var(--easing-out)`.
- Sidebar collapse toggle (button at bottom of sidebar): triggers
  `transition: width var(--duration-normal) var(--easing-out)` on the
  sidebar. Persisted to localStorage.
- "Create new" Button:
  - hover: `bg-button-bg-primary-hover`, transition 100ms.
  - active: `transform: translateY(1px)`.
  - click: opens Modal (per components/modal.tsx pattern).
- Table row hover: `bg-surface`, transition 100ms ease-out.
- Table row click: navigate (or console.log placeholder).
- Sort header click: re-sort + caret rotation 180deg, `transform
  150ms ease-out`.
- Row checkbox click: stops propagation (does NOT trigger row click);
  toggles selection.
- Bulk-action toolbar (appears when ≥1 row selected):
  - Slide down from top of table area: `transform: translateY(-8px → 0)
    + opacity 0 → 1`, both `200ms ease-out`.
  - Buttons inside (variant=ghost size=sm): "Archive", "Delete",
    "Export". Each triggers a Toast (variant=success / error) on
    completion.
- prefers-reduced-motion: cut all `translateY` and `width` transitions
  to 0; keep fades.

**States**

- default: all data populated.
- empty: 0 projects → swap the activity table for the empty-state
  panel from `screens/02-dashboard-empty.png`. KPI tiles still render
  with "—" values.
- loading: 600ms+ → skeleton tiles (Skeleton variant=block) for the
  4 KPIs + 5 skeleton rows (Skeleton variant=line) in the table.
  Sidebar renders normally (assumed cheap).
- error: server error → Toast (variant=error) at top-right with
```
